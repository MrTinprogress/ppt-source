## 案例 1：唤醒词识别率突然下降

标签：性能 profiling / 双核钉核

### PPT 内容

现象：
- 测试时唤醒词识别率从 95% 掉到 60%
- 要说两三次“你好小智”才会响应

怀疑：
- CPU 跑不动？
- 任务被高优先级抢占？

排查工具：
- `vTaskGetRunTimeStats` 打印 CPU 占用
- `vTaskList` 查看任务状态分布

真相：
- `audio_input` 跑在了 Core 0
- 被 WiFi 任务频繁抢占，WiFi 优先级为 23
- 单帧处理延迟从 5ms 飙到 30ms+
- AFE 累积延迟超过容忍阈值

解决：
- 使用 `xTaskCreatePinnedToCore` 显式钉到 Core 1
- 物理隔离 WiFi 抢占

收获：
- 双核不等于自动并行，任务亲和性要主动设计

### 演讲稿

第一个案例是唤醒词识别率突然下降。测试到一半发现唤醒词识别率从 95% 掉到 60%，要说两三次“你好小智”才能唤醒。

最初我怀疑是 CPU 跑不动，于是用 `vTaskGetRunTimeStats` 打印 CPU 占用，发现两个核 IDLE 时间都还有 50% 以上，CPU 不是瓶颈。

接着用 `vTaskList` 看任务状态分布，发现 `audio_input` 任务跑在 Core 0 上，正好和 WiFi 驱动在同一个核。WiFi 优先级 23，远高于 `audio_input` 的优先级 8。每次有网络包，`audio_input` 就会被抢占。单帧音频原本 5ms 能处理完，被抢占后变成 30ms 以上，AFE 累积延迟超过算法容忍阈值，唤醒就漏识别了。

解决方案是用 `xTaskCreatePinnedToCore` 把 `audio_input` 钉到 Core 1，物理隔离 WiFi 抢占。验证后识别率回到 95% 以上。

这个问题让我意识到：双核不等于自动并行，任务亲和性是要主动设计的。

## 案例 2：HA 偶尔控制不动舵机

标签：静默错误排查 / MQTT 重订阅

### PPT 内容

现象：
- HA 面板拖动滑块，舵机有时候不动
- 日志没有任何错误信息
- 重启设备后短时间正常，过一会又复发

怀疑：
- MQTT 消息没收到？
- 回调路由错了？
- 设备和 broker 之间网络问题？

排查工具：
- 在 MQTT `OnMessage` 回调中加日志
- broker 端用 `mosquitto_sub` 订阅同 topic

真相：
- broker 收到了 HA 的消息，但没有转发给设备
- `CleanSession=true` 时，设备断线后 broker 清掉订阅
- 设备自动重连成功，但没有重新订阅 topic
- 消息被 broker 静默丢弃

解决：
- `HaMqttBus` 注册 `OnConnected` 回调
- 每次重连成功，遍历所有 Feature，重订阅全部 topic

收获：
- “没报错”不等于“没出错”，静默丢弃最难发现

### 演讲稿

第二个案例是 HA 偶尔控制不动舵机。这个问题最典型的地方在于：没有任何错误信息，但功能就是不对。

现象是 HA 面板拖动滑块控制舵机，有时候舵机不动。日志干干净净，没报错。重启设备后短时间正常，过一段时间又复发。

我先在 MQTT 的 `OnMessage` 回调里加日志，发现消息根本没进设备。然后在 broker 端用 `mosquitto_sub` 订阅同一个 topic，发现 broker 是收到 HA 消息了，但没有转发给设备。问题就定位到了设备订阅关系上。

根因是项目使用了 `CleanSession=true`。设备断线之后，broker 会清掉它的所有订阅记录。设备虽然自动重连成功了，但没有重新订阅任何 topic。broker 收到 HA 消息时，发现订阅列表里没有这个设备，就把消息静默丢弃了。

解决方案是在 `HaMqttBus` 里注册 `OnConnected` 回调，每次重连成功后遍历所有 Feature 注册过的 topic，全部重新订阅。

这个问题让我形成一个习惯：网络链路里不能只看有没有报错，还要检查失败路径有没有可能被静默吞掉。

## 案例 3：设备运行几小时后重启

标签：内存追踪 / 长期稳定性

### PPT 内容

现象：
- 设备开机正常，跑 3-4 小时后偶发重启
- 重启原因是 Guru Meditation Error
- 单次现象不可复现，但长跑必现

怀疑：
- 内存泄漏？
- 栈溢出？
- 看门狗超时？

排查工具：
- 每 5 秒打印 `heap_caps_get_minimum_free_size`
- `menuconfig` 开启 Stack Overflow Check
- 崩溃地址通过 `idf.py monitor` 自动反汇编定位

真相：
- 某模块每次 WebSocket 重连都会 new 一个对象
- 旧对象没有释放
- `min_heap` 持续下降，每小时下降几 KB
- 3 小时后内存碎片严重，`malloc` 失败并崩溃

解决：
- 重连前显式清理旧资源
- 使用 `unique_ptr` / RAII 管理对象生命周期

收获：
- 长稳问题要看趋势，不是看单次值
- `min_heap` 的历史值比 `free_heap` 当前值更有意义

### 演讲稿

第三个案例是设备运行几小时后重启，这是一个长期稳定性问题。

现象是设备短时间运行都正常，但运行 3 到 4 小时之后会偶发重启。串口里看到的是 Guru Meditation Error，通过 `idf.py monitor` 自动反汇编后，定位到一处 `malloc` 失败。

单看这行代码看不出问题，因为它只是普通内存分配。但“长跑必现”这个特征说明它是累积性问题，要么是内存泄漏，要么是内存碎片。

我加了一段调试代码，每 5 秒打印 `heap_caps_get_minimum_free_size`。这个指标表示开机以来堆内存的历史最低值，比当前空闲内存更能反映系统曾经有多紧张。跑了一个小时后，发现 `min_heap` 持续下降，每小时下降几 KB。

顺着这个方向查代码，发现 WebSocket 模块每次重连都会 new 一个对象，但旧对象没有释放。短时间看不出来，长时间运行后内存碎片越来越严重，最终某次 `malloc` 拿不到连续空间就崩溃。

解决方式是用 `unique_ptr` 和 RAII 管理重连相关资源，确保重连前旧对象一定释放。

这个案例让我意识到：长稳问题要看趋势，不能只看某一刻的值。`min_heap` 的历史最低值，是定位累积性内存问题非常有效的指标。

## 案例 4：设备整体卡死无响应

标签：死锁定位 / mutex 持锁原则

### PPT 内容

现象：
- 某些操作后整个设备无响应
- 按键不响应，串口不打日志
- 不是崩溃，没有 Guru 信息，而是完全静止

怀疑：
- 主任务卡住？
- 死锁？
- 看门狗为什么没触发？

排查工具：
- 重启前抓 `vTaskList`
- 或开启 Task Watchdog，让它自己报出卡住位置

真相：
- `vTaskList` 显示相关任务都是 B（Blocked）状态
- IDLE 任务还在跑，说明 CPU 是空的
- 定位到 `Application::mutex_` 上死锁
- 持锁期间执行了一个 Lambda
- Lambda 内部又调用 `Schedule()`，试图再次拿同一把锁

解决：
- 持锁期间不执行外部回调
- 改成“取数据 → 解锁 → 再执行回调”
- 持锁期间只做最小操作

收获：
- 死锁的典型特征是“业务任务全 Blocked + IDLE 在跑”
- 持锁期间只取数据不做事，是并发安全的核心原则

### 演讲稿

最后一个案例是设备整体卡死无响应，这是一个并发问题。

现象是某些操作之后，整个设备完全静止。它不是崩溃，因为崩溃通常会有 Guru Meditation 信息；它是真的不动了，串口不再打印，按键也没有反应。

这种“看起来死了但没有崩溃”的症状，第一反应应该是死锁。重启前我用 `vTaskList` 抓了一份任务状态，发现所有相关任务都是 B，也就是 Blocked 状态，但 IDLE 任务还在正常跑。这说明 CPU 其实是空的，业务任务都在等锁。

继续定位后发现问题出在 `Application::mutex_`。主任务持着这把 mutex 的时候，遍历执行回调队列里的 Lambda。其中一个 Lambda 内部又调用了 `Schedule()`，而 `Schedule()` 内部也要拿同一把 mutex，于是主任务自己卡在自己持有的锁上，形成了典型的递归锁死锁。

解决方案是改成：先在持锁期间把数据移出来，立刻解锁，然后再执行回调。这样回调内部即使再次调用 `Schedule()`，也不会和原来的锁冲突。

这个问题让我形成一个并发代码原则：持锁期间只做最小操作，只取数据、改数据，不执行任何外部回调。
